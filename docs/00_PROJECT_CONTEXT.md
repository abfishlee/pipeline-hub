# 00. 프로젝트 배경 및 비즈니스 컨텍스트

## 0.1 한 줄 정의

**농축산물 가격 데이터를 다채널에서 수집 → AI 기반 표준코드로 정규화 → 외부 소비자에게 가격 비교/통계 서비스를 제공하는 통합 플랫폼.**

## 0.2 비즈니스 문제

현재 농축산물 가격 데이터는 다음 문제를 갖는다:

1. **유통사·매장별 상품정보체계 상이** — 동일 상품이 "국산 참외 1.5kg", "참외(국내산) 박스", "참외 소(小) 5개입"으로 제각각 표현됨.
2. **가격 정보가 분산** — 대형마트 POS, 온라인몰, SNS 행사 전단, 전통시장, 소비자 영수증 등 출처가 파편화.
3. **품목/등급/포장/판매 단위의 구조화 부재** — 가격 비교·통계 활용 어려움.

따라서 플랫폼은:

1. 다채널 수집 → 원천 보존 → 표준화 → 마스터 통합 → 서비스 제공의 전 단계를 하나의 시스템으로 운영한다.
2. AI로 비정형 상품명을 표준코드에 매핑한다.
3. 수집·변환·반영 이력을 추적 가능하게 유지한다.

## 0.3 수집 채널 (컨셉 이미지 기반)

| 채널 | 방식 | 예상 비중 | 기술 구현 |
|---|---|---|---|
| **대형마트/로컬푸드/SSM POS** | 오픈 API / DB-to-DB | API 50% + DB 5% | FastAPI POST endpoint, DB 증분 수집 |
| **할인 전단 / 행사 SNS** | 이미지 OCR | Crowd 일부 | CLOVA OCR + Crowd 검수 |
| **온라인 채널 (쿠팡/마켓컬리/SSG 등)** | 크롤링 | 5% | Python + Playwright + robots.txt 준수 |
| **aT 보유 전통시장 데이터** | 파일/API | API에 포함 | aT KAMIS 연계 |
| **소비자 영수증** | 모바일 업로드 OCR | **Crowd 대량 (40%)** | CLOVA OCR + 내부 직원 검수 워크플로우 |
| **축산물 — 여기고기 앱** | API 연계 | API에 포함 | 별도 커넥터 |

> **핵심 특징:** Crowd 40%가 "영수증 수집 + OCR 검수" 경로에서 온다. 이는 **사람이 개입하는 비동기 파이프라인**이 필수라는 뜻.

## 0.4 수집 항목

| 필드 | 설명 |
|---|---|
| 품목 (item) | 예: 참외, 쌀, 돼지고기 앞다리살 |
| 등급 (grade) | 특/상/보통 등 |
| 포장 (package) | 박스, 봉지, 낱개 등 |
| 판매 단위 (unit) | 1kg, 5개입, 1박스(10kg) 등 |
| 중량 (weight) | 실제 무게(g 단위 정규화) |
| 당도 (brix) | 과일류 |
| 포장 형태 (pack_type) | 진공포장, 개별포장 등 |
| 가격 (price) | 원 단위 |
| 할인가 (discount_price) | 행사 시 |
| 수집 시각 (observed_at) | 해당 가격이 유효했던 시점 |
| 출처 (retailer / seller) | 어느 유통사/판매자에서 본 가격인지 |

## 0.5 타겟 사용자

| 구분 | 역할 |
|---|---|
| **내부 운영자** | 수집 상태 관제, 파이프라인 설계, SQL 정제, 데이터 품질 검토 |
| **내부 검수자** | Crowd 작업함에서 영수증 OCR 결과 검수, 상품 표준코드 매핑 확인 |
| **승인자** | SQL 정제 결과 Mart 반영 승인 |
| **외부 서비스 소비자 (Phase 4)** | API Gateway를 통해 가격 통계/비교 API 호출. API Key 기반 인증. |

## 0.6 규제/제약

- **개인정보 미수집.** 영수증에서 소비자 식별정보는 마스킹 후 폐기. 매장명/시간/상품/가격만 추출.
- **공공 데이터 표준** (농림축산식품부 표준분류, aT 표준코드) 준수.
- **크롤링 윤리:** robots.txt, 약관, rate limit 준수. 법적 회색 영역 크롤링 금지.

## 0.7 비기능 요구사항

| 항목 | 목표 |
|---|---|
| 수집량 | 평시 100,000 rows/일, 피크 300,000 rows/일 |
| OCR 처리량 | 1,000 페이지/일 (영수증) |
| 신규 가격 → Mart 반영 시간 | 실시간 (수집 완료 후 1분 이내) |
| 크롤링 지연 허용치 | 30분 이내 |
| 가용성 목표 | 99.5% (운영 시간 기준) |
| 장애 복구 | RPO 1시간, RTO 4시간 |
| 보존 정책 | raw 파일 1년(hot 30일/warm 335일), 정형 데이터 영구 |

## 0.8 핵심 설계 원칙

이 프로젝트는 다음 원칙을 위반하지 않는다:

1. **원천 보존** — 원본 파일/JSON/HTML/응답 헤더를 덮어쓰지 않는다. 모든 변환은 재실행 가능해야 한다.
2. **중복 방지** — `content_hash` + `idempotency_key` 2중 방어.
3. **상태 기록** — 모든 수집/변환 작업은 `run.*` 스키마에 상태·입력·출력·파라미터·에러를 남긴다.
4. **게이트 기반 품질** — DQ 실패 시 Mart 반영 자동 HOLD. 승인자만 해제.
5. **Crowd는 비동기** — 사람 개입 노드는 배치 SLA를 깨지 않도록 async-wait 패턴.
6. **외부 서비스 경계** — 외부 소비자는 API Gateway만, 내부 운영자는 Web Portal만 접근.

## 0.9 범위 밖 (명시적 제외)

- 소비자 대상 모바일 앱 자체 개발 (여기고기 앱 연동만)
- 광고/결제 시스템
- 개인정보 기반 추천
- 실시간 경매/가격 알림 (추후 검토)

## 0.10 용어 첫 정의

| 용어 | 정의 |
|---|---|
| **raw** | 수집 직후 원천 그대로의 데이터 (JSON/파일/HTML). |
| **staging (stg)** | 표준 스키마로 변환되었으나 마스터 반영 전 데이터. |
| **mart** | 최종 서비스용 마스터/분석 테이블. |
| **표준코드** | 농림축산식품부/aT 기준 품목 분류 코드. |
| **business_key** | 엔티티별 자연키 (예: 상품은 `retailer_id + retailer_product_code`). |
| **raw_object** | 수집된 원천 1건 단위 (API 응답 1개, 파일 1개 등). |
| **pipeline / pipeline_run / node_run** | Visual ETL 흐름의 정의 / 실행 인스턴스 / 노드 단위 실행. |
| **outbox** | DB 저장과 이벤트 발행을 트랜잭션적으로 묶는 패턴. |
