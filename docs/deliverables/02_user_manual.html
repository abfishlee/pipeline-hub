2. 채널별 공통 — 9단계 프로세스

모든 채널 (POS API / DB / Crawl / OCR) 은 다음 9단계를 따릅니다. 채널별 차이는 각 단계 세부 항목에서 발생.

1

사전 준비 — 인증 + 역할 확인

로그인 화면 → admin 또는 본인 계정으로 로그인. 좌측 사이드바 하단의 본인 역할 (ADMIN/APPROVER/...) 을 확인.

권한 부족 시 메뉴가 안 보이거나 403 에러. 관리자에게 역할 부여 요청.

2

데이터 소스 등록 ADMIN

데이터 소스 → 신규 에서 source_code (예: EMART_API), source_type (API/DB/CRAWL/OCR), 인증 정보 (API key / DB 접속 정보) 입력.

[화면 캡처 자리] 데이터 소스 신규 등록 폼.
캡처 위치: /sources → 우측 상단 "신규" 버튼 클릭
3

(채널별) 첫 수집 트리거 — raw 적재 확인 ADMIN OPERATOR

채널마다 다름:

  • POS API: 수집 작업 → 신규 또는 외부 시스템이 POST /v1/ingest/api 호출
  • DB-to-DB: 데이터 소스 → 해당 source 의 watermark 초기화 + 자동 polling
  • Crawl: 데이터 소스 → 크롤 시작 URL 등록 + Crawler worker 가 자동 처리
  • OCR: 사용자 (앱/웹) 이 POST /v1/ingest/{file,receipt} 로 이미지 업로드

원천 데이터 메뉴에서 채널별 raw_object 가 적재되었는지 확인.

4

워크플로 설계 — Visual ETL Designer OPERATOR APPROVER

Visual ETL Designer 진입 → 좌측 팔레트의 7노드 (NOOP / SOURCE_API / SQL_TRANSFORM / DEDUP / DQ_CHECK / LOAD_MASTER / NOTIFY) 를 캔버스로 drag & drop.

각 노드 클릭 → 우측 패널에서 config_json 편집 (sources / queries / target tables).

노드 간 엣지로 연결 → 상단 저장 (DRAFT 로 영속).

[화면 캡처 자리] Visual ETL Designer 캔버스 — 좌측 팔레트 + 중앙 캔버스 + 우측 노드 설정 패널.
캡처 위치: /pipelines/designer (예: 시드된 retail_api_price__emart 워크플로)
5

SQL 작성 + 승인 — SQL Studio OPERATOR + APPROVER

SQL_TRANSFORM 노드의 SQL 은 안전을 위해 SQL Studio 의 sandbox 에서 사전 검증한다.

  1. SQL Studio → 신규 로 SQL 자산 만들기 (자동 DRAFT v1).
  2. 상단 Validate → sqlglot 로 정적 분석 (참조 테이블 + 위반 사유 노출).
  3. Preview → read-only TX 안에서 LIMIT 1000 자동 부착해 결과 미리보기.
  4. EXPLAIN → PostgreSQL 실행 계획 (FORMAT JSON) 확인.
  5. OPERATOR 가 제출 → DRAFT → PENDING 상태 전이.
  6. APPROVER 가 승인 또는 반려 (self-approval 차단).
  7. APPROVED SQL 만 SQL_TRANSFORM 노드에 연결.
6

PUBLISH (배포) APPROVER

Designer 상단 PUBLISH 버튼. DRAFT 워크플로 자체는 유지되며, 같은 name 안에서 version_no = max+1 인 새 PUBLISHED 워크플로 row 가 생성된다.

그래프(노드/엣지) 가 freeze 되어 향후 DRAFT 가 변경돼도 PUBLISHED 는 영향 없음.

완료 후 화면이 자동으로 새 PUBLISHED 워크플로로 redirect — toast: v2 배포 완료 (release #N).

왜 새 row 인가?

운영자가 PUBLISH 후에도 같은 이름의 워크플로를 계속 편집할 수 있게 하기 위해서. PUBLISHED 그래프는 절대 변하지 않고, DRAFT 는 다음 버전 작업 공간.

상세: ADR-0009 / 배포 이력 메뉴.

7

실행 — 즉시 또는 cron 자동 APPROVER

두 가지 방법:

  • 즉시 실행 — Designer 의 실행 버튼 (PUBLISHED 워크플로 한정). 즉시 pipeline_run 생성 + entry 노드 actor 디스패치.
  • cron 자동 — Designer 툴바의 cron 입력 + "활성" 체크 + "스케줄 저장". 0 5 * * * 같은 5필드 표현식 (UTC). Phase 4 의 Airflow 통합이 polling 후 자동 트리거.
8

실시간 모니터 + 검수 OPERATOR REVIEWER

파이프라인 실행 → workflow / status / 기간 필터 → 상세 진입.

상세 화면 (/pipelines/runs/:runId) 에서 SSE 가 노드별 status 를 실시간으로 색상 (PENDING/READY/RUNNING/SUCCESS/FAILED/SKIPPED) 갱신.

OCR 등에서 confidence 미달 row 는 검수 큐 에 자동 적재 → REVIEWER 가 수동 처리.

[화면 캡처 자리] PipelineRunDetail — React Flow 캔버스 위에 노드 색상 (RUNNING=노랑, SUCCESS=초록).
캡처 위치: /pipelines/runs/[run_id]
9

mart 결과 확인 + 외부 서비스화

SQL Studio 의 mart_freshness_check_24h 같은 템플릿으로 24h 이내 신규 row 수 확인.

외부 소비자에게 노출은 Phase 4 의 /public/v1/prices/* 가 담당. Phase 3 까지는 내부 사용자가 mart 를 SQL 로 직접 조회하거나 Web Portal 의 마스터 메뉴 (Phase 4 추가 예정) 로 확인.

3. 시나리오 A — 유통사 POS API 수집

채널: 유통사 POS API
A-1. 이마트 POS 가격 일배치 (예: retail_api_price__emart)

목표: 이마트 POS API 가 매일 새벽 5시(UTC) 에 자동 호출되어 mart.product_price 로 적재되도록 운영 가능 상태 만들기.

데이터 소스 등록 → Designer 설계 → SQL Studio 승인 → PUBLISH → cron 활성화 → 자동 실행

3.1 작업 순서

  1. 데이터 소스 → 신규: source_code=EMART_API, source_type=API, base_url + API key 등록 ADMIN
  2. 외부 시스템이 POST /v1/ingest/api 로 호출 시작 (또는 수동 테스트 호출).
  3. 원천 데이터 메뉴 → EMART_API 필터 → raw_object 적재 확인 (≥ 1건).
  4. Visual ETL Designer → 신규: 노드 7개 배치 (시드된 retail_api_price__emart 워크플로 그대로 사용 가능).
  5. SQL Studio → 신규: 표준화 SQL 작성 → Validate → Preview → OPERATOR 제출 → APPROVER 승인.
  6. Designer 의 sql_normalize 노드 config_json.sql 에 APPROVED SQL 붙여넣기 → 저장.
  7. PUBLISH → 새 PUBLISHED v2 자동 redirect.
  8. Designer 툴바 cron 영역에 0 5 * * * 입력 + "활성" 체크 + 스케줄 저장.
  9. 다음 날 새벽 5시 KST 14시 자동 실행 (Phase 4 Airflow 통합 후) — Phase 3 한정으로는 "실행" 버튼 수동.

3.2 검증 (다음 날)

  • 파이프라인 실행 → workflow=retail_api_price__emart → 어제 run SUCCESS.
  • SQL Studio → mart_freshness_check_24h Preview → 24h 이내 row 수 ≥ 1.

4. 시나리오 B — DB-to-DB 증분 수집

채널: 외부 PostgreSQL/MySQL incremental
B-1. 외부 ERP 의 상품/가격 테이블 증분 수집

목표: 외부 ERP DB (PG 또는 MySQL) 의 상품/가격 테이블을 watermark 기반 증분 수집해 raw + stg 적재.

DB 접속 정보 등록 → watermark 초기값 → db_incremental worker 활성 → 자동 polling

4.1 작업 순서

  1. 데이터 소스 → 신규: source_code=ERP_DB, source_type=DB, 연결 문자열 (PG/MySQL DSN), watermark 컬럼명 (예: updated_at) 등록.
  2. ctl.data_source.watermark jsonb 에 초기값 (예: {"updated_at": "2026-04-01T00:00:00+00:00"}) 직접 설정.
  3. db_incremental worker (Phase 2.2.7) 가 자동 polling — 매 N분마다 watermark 이후 row 를 가져와 raw.raw_object + stg.price_observation 에 적재.
  4. 이후는 시나리오 A 의 4단계 (Designer) 와 동일 — SOURCE_API 노드의 source_code 만 ERP_DB 로 바꿔 작성.

4.2 watermark 동작

매 polling 후 가져온 마지막 row 의 watermark 컬럼 값을 다음 polling 의 시작점으로 저장. CDC (Change Data Capture) 시 before/after 모두 raw 에 보관.

주의: 외부 DB 부하

watermark 컬럼에 인덱스가 있어야 외부 DB 부하가 작다. 5분 간격 polling 으로 시작해 부하 보면서 조정.

5. 시나리오 C — 온라인 크롤링

채널: HTML 크롤링 (쿠팡 / 마컬)
C-1. 쿠팡 상품 페이지 시간배치 (예: online_crawl__coupang_hourly)

목표: 쿠팡의 상품 페이지를 매시간 정각 크롤링해 가격 변동 추적.

데이터 소스 (URL 셋) → Crawler worker → robots.txt 검증 → content_hash dedup → stg 적재

5.1 작업 순서

  1. 데이터 소스 → 신규: source_code=COUPANG_CRAWL, source_type=CRAWL, 크롤 시작 URL 셋 + 파서 버전 등록.
  2. Crawler worker (Phase 2.2.8) 가 자동 크롤 — robots.txt 준수, content_hash 로 같은 페이지 dedup.
  3. HTML → 정형 row 파싱 → stg.price_observation.
  4. Designer 워크플로 — 시드된 online_crawl__coupang_hourly 그대로 사용 가능 (cron 0 * * * * 매시간).

⚠ robots.txt 준수 필수

Crawler worker 는 robots.txt 의 Disallow 규칙을 자동 검증하고 차단된 URL 은 즉시 RobotsBlocked 예외. 사이트 운영자와의 약속 준수.

5.2 가격 이상치 (BLOCK 정책)

크롤링은 파서 버그로 가격 단위가 잘못 잡히는 경우가 있어 online_crawl__coupang_hourly YAML 의 dq_check_price_outliers 노드는 severity=BLOCK — 가격 ≤ 0 또는 5천만원 초과 row 발견 시 pipeline_run 즉시 FAILED. (Phase 4 의 ON_HOLD 흐름과 결합 예정.)

6. 시나리오 D — OCR (영수증 / 전단)

채널: OCR (영수증 / 마트 전단)
D-1. 소비자 영수증 OCR 자동 처리 + Crowd 검수

목표: 사용자가 앱/웹에서 영수증 사진을 업로드하면 OCR 로 추출 → confidence ≥ 0.85 자동 반영 / 미만은 Crowd 검수 큐.

사용자 업로드 → CLOVA OCR → Upstage 폴백 → confidence 게이트 → 자동 반영 / Crowd

6.1 사용자 측 (앱/웹)

  1. 로그인 후 영수증 업로드 화면 (앱) → 사진 촬영 → POST /v1/ingest/receipt.
  2. multipart/form-data 로 이미지 업로드 (≤ 10MB).
  3. 처리 결과는 비동기 — 사용자에게 "접수 완료" 알림 즉시 표시.

6.2 자동 처리 (백엔드)

  1. raw.raw_object 적재 (RECEIPT_IMAGE) + Object Storage 에 원본 사진.
  2. OCR worker — CLOVA OCR 호출 (1차) → 실패 시 Upstage 폴백 (2차).
  3. 추출된 매장명/품목/가격을 stg.price_observation 에 INSERT.
  4. confidence ≥ 0.85 인 row 는 자동 표준화 (3단계 fallback) → mart 반영.
  5. confidence < 0.85 또는 표준화 실패 row 는 run.crowd_task 에 적재.

6.3 검수자 측 REVIEWER

  1. 검수 큐 진입 → reason 별 필터 (ocr_low_confidence / std_low_confidence / price_fact_low_confidence / sample_review).
  2. 해당 row 클릭 → 원본 영수증 사진 미리보기 + OCR 추출 결과 + 후보 std_code top-5.
  3. 승인 / 반려 마킹. (Phase 3 한정으로는 status 전이만 — Phase 4 부터 mart 자동 반영 + 이중 검수.)

6.4 개인정보 마스킹

⚠ 영수증 PII 마스킹 필수

전화번호 / 카드 끝자리 / 주소는 OCR 단계에서 자동 정규식 마스킹 (Phase 2.2.4). 원본 사진은 Object Storage 에 암호화 저장. 외부 API 응답에 절대 포함 금지.

7. 추가 — 스케줄 / Backfill / 재실행

7.1 스케줄 cron 설정

  1. PUBLISHED 워크플로의 Designer 진입.
  2. 툴바의 cron 입력란 (5필드, UTC 기준): 분 시 일 월 요일.
  3. "활성" 체크박스 + 스케줄 저장.
  4. 완료. 다음 cron 발화 시각부터 자동 실행 (Phase 4 Airflow 통합 후).
UTC cron의미 (한국시간)예시 시나리오
0 5 * * *매일 14:00 KSTPOS API 일배치
0 2 * * *매일 11:00 KST영수증 OCR 일배치
0 * * * *매시간 정각크롤링 시배치
*/15 * * * *매 15분DB-to-DB 증분 polling

7.2 Backfill — 과거 일자 재처리

  1. PUBLISHED 워크플로 Designer → Backfill 버튼.
  2. 시작/종료 날짜 입력 (예: 2026-04-01 ~ 2026-04-07).
  3. 실행 → 일자별 PENDING pipeline_run 1개씩 생성 (멱등 — 이미 있으면 기존 ID 반환).
  4. 최대 366일 한도. 그 이상은 chunk 로 나눠서.

7.3 재실행 — 처음부터 또는 특정 노드부터

  1. 파이프라인 실행 → 해당 run 상세.
  2. 상단 처음부터 재실행 — 새 pipeline_run + entry 노드 READY.
  3. 각 노드 row 의 이 노드부터 — 해당 노드의 ancestors 는 SUCCESS 시드, 노드 자체는 READY, 후손은 PENDING.

언제 "이 노드부터" 를 쓰나?

SQL_TRANSFORM 의 SQL 만 고쳤을 때, 앞 단계 (SOURCE_API / DEDUP) 를 다시 돌릴 필요가 없으면 SQL_TRANSFORM 노드부터 재실행 — 시간/리소스 절약.

8. 추가 — SQL Studio 승인 라이프사이클

8.1 self-approval 차단

OPERATOR 가 submit 한 SQL 을 본인이 approve 하려고 하면 422 거부. APPROVER 도 동일 — 본인이 submit 한 PENDING 은 다른 APPROVER 가 처리해야 함. 이중 검토를 강제하는 보안 정책.

8.2 SQL 자산을 SQL_TRANSFORM 노드와 연결

  1. SQL Studio 에서 새 SQL 자산 작성 → APPROVED 까지 진행.
  2. Designer 의 SQL_TRANSFORM 노드 클릭 → config_json 의 sql 필드에 APPROVED SQL 본문 붙여넣기 (Phase 3 한정 — Phase 4 에서 자산 ID 직접 참조).
  3. Designer 의 하단 SQL Studio 영역의 Validate 통과하면 노드 config 저장 OK.

9. 트러블슈팅

9.1 로그인 / 권한

증상원인 / 해결
로그인 401 비밀번호 오타 / 사용자 비활성. ADMIN 에게 비활성 해제 요청.
메뉴가 안 보임 역할 부족. 사이드바 하단의 본인 역할 확인.
버튼 클릭 시 403 API 권한 미달. 같은 메뉴라도 일부 액션은 더 높은 역할 필요 (예: SQL approve = APPROVER).

9.2 Visual ETL Designer

증상원인 / 해결
저장 시 422 node_key 중복 / 엣지가 없는 node_key 참조 — 우측 패널의 node_key 정합성 확인.
PUBLISH 시 409 노드 0개 워크플로. 최소 1개 노드 필요.
실행 버튼이 비활성 워크플로 status 가 DRAFT — 먼저 PUBLISH.
편집이 안 됨 PUBLISHED 워크플로는 readonly. 좌측 사이드바의 같은 name 의 DRAFT 워크플로로 이동해 편집.

9.3 SQL Studio

증상원인 / 해결
Validate 실패: schema not allowed mart/stg/wf 외 schema 사용. SQL 수정.
Validate 실패: only SELECT statements DROP/UPDATE/DELETE/COPY 등 거부. SELECT 만.
Preview 실패: statement_timeout SQL 이 30초 초과. WHERE 추가 / 인덱스 활용 / LIMIT.
Approve 시 422 self-approval — 다른 APPROVER 에게 요청.

9.4 파이프라인 실행

증상원인 / 해결
run 이 PENDING 에서 안 진행 Worker 미가동. Runtime 모니터 에서 worker 백로그 확인.
NOTIFY 노드 SUCCESS 인데 알림 안 옴 Phase 3 한정 — outbox 만 적재, 실 발송은 Phase 4 notify_worker.
같은 (workflow, run_date) 가 두 번 발생 안 함 — backfill / cron 모두 멱등 (이미 있으면 기존 ID 반환).

9.5 OCR / Crowd

증상원인 / 해결
OCR 결과가 모두 이상 이미지 회전/품질 문제. 사용자에게 재업로드 요청.
검수 큐가 끝없이 쌓임 confidence 임계 0.85 가 너무 엄격할 수 있음. ADR-0005 의 임계값 보정 검토 (Phase 4).
검수 승인 후 mart 안 들어감 Phase 3 한정 — Crowd 결과 mart 자동 반영은 Phase 4 의 정식 모듈.

9.6 시스템 전반

증상원인 / 해결
500 Internal Server Error /readyz 확인 → DB / Object Storage 연결 점검. ADMIN 에게 알림.
SSE 끊김 (실시간 색상 안 바뀜) 5초 후 자동 재연결 + 5초 폴링 fallback. 1분 뒤에도 안 되면 새로고침.
대시보드 메트릭 로딩 안 됨 Prometheus 미가동. Runtime 모니터 → /metrics 직접 호출 시도.
